The Bitap Algorithm

The design decision: use bit-parallelism to make approximate matching fast

The problem this lesson solves

The scoring DP from Lessons 5 and 5b runs in O(|query| × |candidate|) time — polynomial, but still non-trivial when you have thousands of candidates. The subsequence gate filters most of them, but a different class of problems needs something faster still: scanning large bodies of text for approximate matches, without any pre-filtering.

Design decision Represent the matching state as a bitmask, where each bit tracks whether a particular prefix of the pattern has been matched ending at the current text position. A single bitwise operation advances all states at once, exploiting the CPU's 64-bit word to do 64 comparisons in one instruction. This is the Bitap algorithm (also called Shift-Or or Shift-And), introduced by Baeza-Yates and Gonnet (1992).

The NFA intuition

Before bitmasks, think of matching as a non-deterministic finite automaton (NFA). For pattern cat, the NFA has one state per pattern character plus a start state:

The NFA starts in state 0. Each time a text character matches the expected pattern character, the NFA moves to the next state. Multiple states can be active simultaneously — this is the non-determinism. If we ever reach the final state, we found a match.

The bitmask insight Instead of tracking which states are active in a set, encode them as a single integer: bit i is 1 if state i is active. One bitwise operation advances all active states at once.

The Shift-And algorithm (exact matching)

Precomputation: for each character c in the alphabet, build a mask D[c] where bit i is 1 if pattern[i] == c.

Scanning: maintain a state bitmask R (initially 0). For each text character t:

// Shift all active states forward by one, open a new state 0, keep only
// states where the pattern character matches the current text character.

R = ((R << 1) | 1) & D[t]

// If bit (m-1) is set in R, the full pattern was just matched.
if (R & (1 << (m-1))) → match found

Each line decodes as:

  • R << 1 — advance every active state by one position in the pattern
  • | 1 — always try starting a fresh match at position 0
  • & D[t] — discard any state where the pattern character does not match the text character

Extending to k errors (approximate matching)

Exact matching is useful, but fuzzy search needs tolerance for errors. The Bitap algorithm extends cleanly: instead of one bitmask, maintain k + 1 bitmasks, one per error level.

// R[j] = states reachable with exactly j errors
for j = k downto 1:
  R[j] = (
    ((R[j] << 1) | 1) & D[t] // match (same as exact)
    | (R[j-1] << 1) // deletion from text
    | R[j-1] // insertion into text
    | (R[j-1] << 1) & ~D[t] // substitution
  )
R[0] = ((R[0] << 1) | 1) & D[t] // exact match layer

Each layer adds a one-step "rescue" from the layer below it. A state that would have died (wrong character) in layer j−1 gets propagated to layer j with one more error counted.

Mapping to edit operations
  • R[j-1] << 1 (no AND with D[t]) — deletion: skip a text character
  • R[j-1] — insertion: advance in pattern without consuming text
  • (R[j-1] << 1) & ~D[t] — substitution: consume text but allow mismatch

Bitap vs. the fzy scoring DP

PropertyBitap (Shift-And/Or)fzy scoring DP
Matching modelEdit distance (k errors)Subsequence
OutputMatch positions (yes/no + where)Score (for ranking)
Time per candidateO(m·n / w) — bit-parallelO(m·n)
Pattern length limit~64 chars (one CPU word)No hard limit
Ranking supportNo — binary resultYes — continuous score
Used byagrep, fzf (inner loop)fzy, VS Code picker
They solve different halves of the problem Bitap excels at the matching question: is this candidate within k edits of the pattern, and where? The fzy scoring DP excels at the ranking question: among candidates that match, which one did the user most likely mean? Production fuzzy finders often combine both.
