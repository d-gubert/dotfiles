The Problem: Why Exact Search Fails

Understanding what we are trying to solve before we solve it

The promise of exact search

The simplest search algorithm imaginable is a direct string comparison: does the candidate contain the query as a substring? This is what str.includes(query), grep, and LIKE '%…%' all do.

It is fast, predictable, and sufficient — until the moment a human is doing the typing.

Core insight Exact match assumes the query is a precise, correctly-spelled substring of the target. Human queries almost never are.

Three ways humans fail to type what they mean

1. Typos and transpositions

A developer types comit instead of commit. They type recieve instead of receive. Exact match returns nothing. But the intent is unambiguous.

2. Abbreviations

A user types usr/lcl to find /usr/local/bin. They type MC to find MyComponent.tsx. No substring match exists, but the intent is clear: the query characters all appear in the target, in order.

3. Fuzzy recall

A user remembers a file is "something about authentication, starts with auth, has config in the name." They type authcfg. The file is named auth_config_loader.ts. Again: exact match fails, human intent succeeds.

What the problem really is

The challenge has two distinct parts, and most confusion about fuzzy search comes from conflating them:

  1. Matching — deciding whether a candidate is "close enough" to the query at all.
  2. Ranking — when multiple candidates pass the threshold, deciding which one the user probably meant.

These require different machinery. A good fuzzy search algorithm solves both — and the rest of this course shows how each design decision targets one or the other.

Preview of what's coming Edit distance solves matching: it gives us a number representing how "far apart" two strings are. Scoring models (gap penalties, consecutive bonuses, boundary bonuses) solve ranking: they let us order results by likely intent.

What properties must a solution have?

Before building anything, it helps to list the requirements the solution must satisfy. A fuzzy search algorithm should:

  • Tolerate small typos — off-by-one character edits should not disqualify a match.
  • Support abbreviations — all query characters appear in the target, in order (subsequence matching).
  • Rank better matches first — comit → commit should rank above comit → recommit_changes_v2.log.
  • Prefer compact matches — matching 4 consecutive characters should score higher than the same 4 characters scattered across a 200-character string.
  • Prefer early and boundary matches — a match at the start of a word or path component feels more intentional than one buried in the middle.
The tradeoff Every relaxation of exact match increases the number of results returned (recall goes up). The scoring model is what keeps precision usable by surfacing the right result first.
