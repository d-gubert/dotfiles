Computing Edit Distance: The DP Table

The design decision: use dynamic programming to find the minimum in polynomial time

The problem this lesson solves

We know edit distance is the right metric. But how do you compute it? The naïve approach — try every possible sequence of edits and keep the shortest — has exponential time complexity. For a 10-character query against a 50-character filename, that's trillions of paths to evaluate.

Design decision Observe that the problem has overlapping subproblems: the edit distance between the first 4 characters of A and the first 3 characters of B is a subproblem of the full computation — and it gets used many times. Store those subproblem answers in a table and look them up instead of recomputing them. This is dynamic programming. Time complexity drops from exponential to O(m × n).

The recurrence relation

Define dp[i][j] as the edit distance between the first i characters of string A and the first j characters of string B.

There are only ever three cases when filling a cell:

If A[i] == B[j]:          dp[i][j] = dp[i-1][j-1]          // no edit needed

If A[i] != B[j]:          dp[i][j] = 1 + min(
                              dp[i-1][j-1],  // substitute
                              dp[i-1][j],    // delete from A
                              dp[i][j-1]     // insert into A
                           )

The base cases initialise the first row and column: converting an empty string to a k-character string costs exactly k operations.

Why three predecessors? Each cell in the table represents a prefix-pair. To get here, either:
  • The last characters matched (come from the diagonal — free move)
  • We deleted a character from A (come from above — cost 1)
  • We inserted a character into A (come from the left — cost 1)
The substitute case is the diagonal with a cost of 1 when characters differ.

Reading the finished table

The answer is always in the bottom-right cell: dp[m][n], where m and n are the lengths of the two strings. The path from bottom-right back to top-left (tracing the minimum choices) tells you exactly which edits were made — a full alignment of the strings.

For fuzzy search, we only need the number in that final cell. Computing the full path (the alignment) is an optional extra step used when you want to highlight which characters matched.

Complexity

ApproachTimeSpace
Naïve recursion (no memoisation)O(3m)O(m)
DP table (Wagner–Fischer)O(m × n)O(m × n)
DP table, two-row optimisedO(m × n)O(n)
Space optimisation Each row only needs the previous row's values, so you can discard all but the last two rows. This reduces space from O(m × n) to O(n) — important when scanning many candidates.
