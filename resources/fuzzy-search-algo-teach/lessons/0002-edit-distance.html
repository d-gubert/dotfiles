Edit Distance: Formalising "Closeness"

The design decision: define a precise metric for how far apart two strings are

The problem this lesson solves

In Lesson 1 we established that fuzzy search needs to decide whether a candidate is "close enough" to the query. But close enough is not an algorithm — it's a feeling. To build a machine that makes this decision reliably, we need a number.

Design decision Define a distance metric between two strings: a single integer that captures how many primitive editing steps are needed to transform one string into the other. This metric is called edit distance (or Levenshtein distance, after Vladimir Levenshtein, 1965).

Once we have this number, matching becomes a threshold check: "accept the candidate if its edit distance from the query is ≤ k." The choice of k is still a policy decision, but the distance itself is objective and computable.

The three primitive operations

Edit distance counts the minimum number of single-character edits needed to turn string A into string B. The allowed edits are:

OperationExampleCost
Insert a charactercat → cart1
Delete a charactercomit → cmit1
Substitute a characterkitten → sitten1
Why these three? These are the three irreducible ways a single character can be wrong. Insert covers missing characters; delete covers extra characters; substitute covers wrong characters. Together they span all single-character divergences between two strings.

Calculating edit distance by hand

Let's find the edit distance between comit (a typo) and commit (the intended word). 

comit   →   commit

Step 1: comit → commit    (insert 'm' after the first 'm')
                           cost: 1

Total edit distance: 1

Now a harder example: kitten → sitting.

kitten  →  sitting

Step 1: kitten  → sitten   (substitute 'k' → 's')
Step 2: sitten  → sittin   (substitute 'e' → 'i')
Step 3: sittin  → sitting  (insert 'g' at the end)

Total edit distance: 3
The key word is "minimum" There are many sequences of edits that transform one string into another. Edit distance is the minimum such count. Finding this minimum naively (by trying all paths) is exponentially expensive — which is exactly the problem Lesson 3 solves with dynamic programming.

Why the minimum matters for fuzzy search

We want edit distance to represent the most charitable interpretation of the user's intent. If a human could have produced the query in one keystroke slip, it should cost 1 — not 5. Using the minimum edit count correctly captures the "how many mistakes did this take?" intuition.

A note on variants

Levenshtein distance (insert, delete, substitute) is the most common formulation, but two variants are worth knowing:

  • Damerau–Levenshtein — adds a fourth operation: transposition (swapping two adjacent characters, e.g. teh → the). This catches a very common class of typo. Many real fuzzy search implementations use this variant.
  • Hamming distance — only counts substitutions, and requires strings of equal length. Simpler, but too rigid for search.
Design insight The choice of which operations to allow is itself a design decision. Levenshtein without transposition will give teh a distance of 2 from the (delete + insert). With Damerau's extension, it's 1. The right choice depends on the error model for your users.
