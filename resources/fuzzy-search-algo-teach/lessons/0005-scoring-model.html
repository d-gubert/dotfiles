The Scoring Model

The design decision: rank matches by intent, not just by existence

The problem this lesson solves

After the subsequence gate (Lesson 4), we know a query's characters all appear in the candidate in order. But a query like cfg matches cfg.ts and app_configuration_manager.ts equally — both are valid subsequences. How do we know which one the user meant?

Design decision Assign each match a score based on the quality of the alignment, not just its existence. The scoring model encodes human intuitions about intentionality: consecutive runs are more deliberate than scattered letters; matching word starts shows structural awareness; compact matches in short strings are more precise.

This is the heart of fzy's algorithm. Every design decision in the scoring model solves one specific failure of naive subsequence matching.

The three scoring decisions

Decision 1 — Consecutive match bonus

If matched characters run consecutively in the candidate, the user almost certainly typed exactly that substring. Award a large bonus per consecutive pair.

Scattered match
configuration
query cfg matched to positions 0, 3, 5 — gaps between every char
Consecutive match ✓ wins
cfg.ts
query cfg matched to positions 0, 1, 2 — three consecutive characters

Decision 2 — Word boundary bonus

Matching the first character of a word (after /, _, -, space, ., or uppercase-after-lowercase) means the user is thinking in terms of structure, not raw characters. Award a bonus scaled by boundary type.

Boundary typeExample triggerBonus
Path separator //usr/local+0.9
Word char _ - spaceauth_config+0.8
camelCase capitalmyComponent+0.8
Dot separatorindex.ts+0.6
No boundaryconfig+0.0
Why these values? Path separators rank highest because typing usr/l precisely signals you want something inside /usr/. Word chars are next — they show you know the token structure. Mid-word matches are penalised because they could easily be accidental.

Decision 3 — Gap penalty (affine)

Every unmatched character between two matched characters is a "gap." Gaps are penalised because they suggest the match was accidental. fzy uses an affine gap penalty: a fixed cost for opening a gap, plus a smaller cost per additional character in it. 

gap_cost(length) = GAP_OPEN + (length - 1) × GAP_EXTEND

where: GAP_OPEN = -0.01   (per character, inner gaps)
       Outer gaps (leading/trailing) = -0.005 per character
Why affine and not linear? Borrowed from bioinformatics (Gotoh, 1982). A single long gap is more "forgivable" than many short gaps: one long skip might just mean the file has a long suffix. Many small gaps suggest the match is scattered by coincidence. Affine penalties encode this by making the first character of a gap more expensive than subsequent ones.

The two-matrix DP

The scoring DP builds two matrices over the same (query × candidate) grid:

  • M[i][j] — the best score for matching the first i query characters to the first j candidate characters, with query[i] matched to candidate[j]. May have arrived here via a gap.
  • D[i][j] — the same, but query[i] is part of a consecutive run (no gap before this match). Used to compute the consecutive bonus correctly. 
// Bonus at candidate position j
B[j] = boundary_bonus(candidate, j)

// D: consecutive-run score ending at (i, j)
D[i][j] = max(
  M[i-1][j-1] + CONSECUTIVE + B[j],   // extend a run from any previous match
  D[i-1][j-1] + CONSECUTIVE + B[j]    // extend a confirmed consecutive run
)

// M: best score overall ending at (i, j)
M[i][j] = max(
  D[i][j],                             // end a consecutive run here
  best over k < j: M[i-1][k] + gap_cost(j - k) + B[j]
)
Why two matrices? The consecutive bonus applies when both the previous and current query characters are matched without a gap. If you tracked only one matrix you couldn't distinguish "I arrived here from a consecutive run" from "I arrived here via a gap." The D matrix carries that information.
