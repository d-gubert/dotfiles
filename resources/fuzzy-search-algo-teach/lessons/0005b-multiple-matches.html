Competing Alignments

When a query can match in multiple places, how does scoring choose?

The hidden complexity of subsequence matching

Lesson 5 described the scoring rules in isolation. But there is a subtlety we glossed over: for a given query and candidate, there may be many valid subsequence alignments — many different ways to assign each query character to a candidate position.

The scoring DP does not pick one alignment upfront. It evaluates all of them simultaneously by filling the M and D matrices, and the winner emerges from the recurrence naturally.

The motivating example Query ts, candidate timestamp.ts.

The characters t and s both appear multiple times. t is at positions 0, 5, 10 — and s is at positions 4 and 11. That gives four valid alignments. Which one should win?

Enumerating the alignments

Let's lay out every valid alignment for ts in timestamp.ts and score each one manually before looking at what the algorithm picks.

The alignment that matches .ts at the end wins — by a large margin. The consecutive bonus (+1.0) and dot boundary bonus (+0.6) together dominate the leading gap penalty (−0.05 for 10 skipped chars).

This is the right answer A user who types ts in a file picker almost certainly wants TypeScript files ending in .ts. The scoring model reaches this conclusion automatically — not through hardcoded file-extension logic, but through the interaction of the consecutive bonus and boundary bonus.

Why the DP finds this without enumerating explicitly

The matrix recurrence means the DP "tries" all alignments at once. At each cell M[i][j], it considers:

  • Arriving consecutively from the previous match (via D[i-1][j-1])
  • Arriving after a gap from the best earlier match (via prevScore)

The cell simply keeps whichever is larger. No explicit enumeration. By the time we reach the last query character, the cell with the highest value represents the best alignment — whatever it turned out to be.
