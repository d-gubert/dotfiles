From Distance to Search

The design decision: choose the right matching model for your query type

The problem this lesson solves

We can now compute edit distance. But a distance number alone is not a search algorithm. We need to decide: which candidates are matches? And we need to do it fast enough to run on every candidate in a large list.

This lesson covers two design decisions made at this junction — and one key pivot that explains why tools like fzy and fzf use a different matching model than edit distance entirely.

Approach 1: Fixed threshold k

The obvious first step: accept a candidate if its edit distance from the query is ≤ k. Choose k based on how permissive you want to be.

Fixed-threshold demo

Query: cfg — drag the threshold and watch which candidates pass.

The problem with fixed k At k=2, cfg.ts (dist 3) is rejected while xx (dist 3 too) passes because it is shorter. The same absolute distance means very different things depending on candidate length. A distance of 3 on a 4-character string is devastating; on a 40-character string it barely matters.

Approach 2: Normalised distance

Divide the edit distance by the length of the longer string. This gives a similarity ratio between 0 (identical) and 1 (completely different). 

normalised_distance(A, B) = edit_distance(A, B) / max(len(A), len(B))

A threshold of 0.3 means "accept if fewer than 30% of characters differ." This scales gracefully with string length.

Better, but not sufficient Normalised distance solves the length problem for typo-style queries. But it still fails for abbreviation queries like authcfg → auth_config_loader.ts, where edit distance is large (11) but the user's intent is obvious. The characters are all there — just not contiguous.

The pivot: subsequence matching

This is the key insight behind fzy, fzf, and most modern command-palette and file-finder tools:

Design decision For interactive file/command search, the user is almost always abbreviating, not misspelling. They type the first letters of words, or the distinctive characters. The right matching model is therefore subsequence matching — not edit distance.

A query Q is a subsequence of candidate C if all characters of Q appear in C in the same order (but not necessarily contiguously).

This is a stricter gate than edit distance in one sense (you must have all characters, in order) but far more permissive in another (gaps are free — you can skip any number of characters in the candidate).

Edit distance gives large, meaningless numbers for abbreviation queries. Subsequence gives a clean binary: either all your characters are present in order, or they aren't.

The subsequence check algorithm

Checking whether Q is a subsequence of C is a simple greedy scan in O(|C|) time — much faster than the O(|Q| × |C|) DP table: 

function isSubsequence(query, candidate):
  qi = 0                       // pointer into query
  for each character c in candidate:
    if c == query[qi]:
      qi++                     // advance query pointer on match
    if qi == query.length:
      return true              // all query chars found in order
  return false                 // ran out of candidate chars
Why this matters for performance In a fuzzy finder scanning thousands of filenames, the subsequence check runs first. Only candidates that pass it proceed to the expensive scoring step. Most candidates are eliminated in O(n) time, making the overall system feel instant.

The complete matching pipeline

Putting it together, here is the pipeline that fzy-style tools use:

  1. Subsequence gate (O(n) per candidate) — if the query is not a subsequence of the candidate, reject immediately. No score computed.
  2. Score (O(|Q| × |C|) per surviving candidate) — run the scoring DP to assign a rank. This is what Lesson 5 covers.
  3. Sort by score — the highest-scoring candidate is the top result.
The key shift fzy does not use edit distance at all. The subsequence check replaces the threshold step entirely. The scoring model (next lesson) replaces the "which edit is cheapest?" question with "which match position feels most intentional?"
